## Answer Key and Rationale

**Q1 — B**
ISO/IEC/IEEE 15288:2023 §6.2 defines exactly six stages: Concept, Development, Production, Utilisation, Support, and Retirement, in that order. Options A, C, and D describe software development phases or project management constructs, not the ISO 15288 life cycle stages.

**Q2 — B**
The SE value proposition (SEH5 Ch1 and Ch2) holds that early requirements investment reduces total life cycle cost by preventing expensive late-stage changes. The cost-of-change curve (Boehm, cited in SEH5 Ch2) shows exponential cost increase with life cycle progression. Options A, C, and D are either inaccurate or describe secondary, procedural reasons rather than the fundamental SE argument.

**Q3 — C**
ISO 15288 §6.2 and SEH5 Ch2 define a decision gate as a formal review point assessing exit criteria readiness and authorising (or denying) stage progression. Options A, B, and D describe tools, documents, or contractual mechanisms — none of which captures the ISO 15288 definition.

**Q4 — B**
"Conditional Proceed" is the appropriate outcome when most criteria are met but defined open items remain — provided those items are tracked with owners and close-out dates. "Return" is reserved for when the stage is materially incomplete. "Terminate" is reserved for fundamental programme viability failure. (SEH5 Ch2; ISO 15288 §6.2)

**Q5 — B**
ISO 15288 defines no process called "Development." The work of developing a system is performed by the fourteen Technical Processes (e.g., System Requirements Definition, Architecture Definition, Design Definition, Implementation, Integration, Verification, Validation) invoked within the Development Stage and, where applicable, in other stages. (ISO 15288 §6.4; SEH5 Ch2)

**Q6 — B**
The Verification process (ISO 15288 §6.4.9) is a Technical Process applicable in any stage where it is relevant. In Production, it verifies that manufactured instances conform to the approved configuration. In Support, it verifies the system after maintenance. The process is not stage-locked. (ISO 15288 §6.4; SEH5 Ch2 and Ch3)

**Q7 — C**
Medical device regulatory frameworks (e.g., FDA QSR, MDR) require documented, sequential phase reviews with formal sign-off — a classic use case for the sequential life cycle model. Iterative and concurrent models are unsuitable where regulatory bodies require each phase to be formally approved before proceeding. (SEH5 Ch2)

**Q8 — B**
Commencing production tooling procurement before CDR is a concurrent model decision — it compresses schedule but accepts the risk that post-CDR design changes will require tooling rework. This is the primary SE risk of concurrency: upstream change cascades into downstream work that has already begun. (SEH5 Ch2)

**Q9 — B**
The system life cycle (ISO 15288) is an engineering construct describing a system's evolution from concept to retirement. The product life cycle is a commercial/marketing construct (introduction, growth, maturity, decline). ISO 12207 governs software life cycle processes, not the product life cycle per se. (SEH5 Ch2)

**Q10 — B**
ISO 15288 §6.2 mandates the principle of decision gates with exit criteria. It does not prescribe named reviews. SRR, PDR, CDR, ORR, etc. are programme-specific implementations. (ISO 15288 §6.2; SEH5 Ch2)

**Q11 — C**
The cost-of-change research (Boehm, cited in SEH5 Ch2) shows exponential increase — errors found in operations can cost 100–1,000x more than the same correction in early stages. Option D's specific "10x factor" is not an ISO 15288 specification; the curve is empirically derived, not formulaic.

**Q12 — C**
The Agreement Processes in ISO 15288 §6.1 govern acquirer-supplier relationships — specifically the Acquisition process (acquirer perspective) and the Supply process (supplier perspective). (ISO 15288 §6.1)

**Q13 — C**
System integration and qualification testing (verification of the design and validation against stakeholder needs) are Development Stage activities. Production Stage testing verifies conformance of manufactured instances, not the design. (ISO 15288 §6.2; SEH5 Ch2)

**Q14 — C**
Delivering the system in three successive, usable capability releases is the Evolutionary (Incremental) model. Iterative would imply the same release is refined repeatedly. Sequential would imply one final delivery. (SEH5 Ch2)

**Q15 — C**
Detailed system requirements analysis for a *replacement* system is a Concept Stage activity for the *new* system — not a Retirement Stage activity for the existing system. Retirement activities are decommissioning, disposal, and archiving. (ISO 15288 §6.2; SEH5 Ch2)

**Q16 — B**
Total Life Cycle Cost (LCC) is the correct metric for whole-life system selection. Acquisition cost is only 20–30% of LCC; the dominant costs occur in operations and support. Minimising acquisition cost or schedule at the expense of LCC is poor SE decision-making. (SEH5 Ch2)

**Q17 — C**
ISO/IEC/IEEE 15288:2023 defines 30 processes: 14 Technical + 8 Technical Management + 2 Agreement + 6 Organisational Project-Enabling = 30. (ISO 15288 §6)

**Q18 — B**
System-level verification is a Technical Process (ISO 15288 §6.4.9) operating at the system level. Software sprint reviews verify software functionality, not system-level requirements. Agile SE preserves the SE layer above software development methods; it does not replace system-level V&V. (ISO 15288 §6.4.9; SEH5 Ch2 and Ch3)

**Q19 — C**
Option C is incorrect. Architecture optimisation is a Development Stage activity. In the Production Stage, the system architecture is already frozen; activities focus on manufacturing, assembly, and conformance testing against the approved configuration — not designing or optimising the architecture. (ISO 15288 §6.2; SEH5 Ch2)

**Q20 — B**
ISO 15288 defines processes independently of stages. A programme's chosen life cycle model determines which processes are applied at each stage, and the same process can be applied in multiple stages. Option A is incorrect (Technical Processes are not limited to the Development Stage). Options C and D are factually wrong. (ISO 15288 §6; SEH5 Ch2)
